# Decision Log

## 2026-05-20: 앱 정식 명칭 확정 — 설교필기 (PulpitInk)

결정:
- 한글 정식 명칭: **설교필기**
- 영문 정식 명칭: **PulpitInk**
- 브랜드 컨셉: "설교단(Pulpit)의 잉크(Ink)" — 설교 특화 데스크톱 STT 앱

근거:
- 5개 후보(말씀필사/VoiceQuill, 소리글/SoriScript, 음성기록관/ScribeFlow, 설교필기/PulpitInk, 하늘소리/CelestialVox)에 대해 브랜드 충돌 조사를 실시했습니다.
- VoiceQuill(iOS STT 앱), 소리글AI(회의 녹음 앱), ScribeFlow(5개 이상 서비스), CelestialVox(Google Play 기도문 앱)는 동일/유사 도메인에서 활성 브랜드가 존재하여 탈락.
- PulpitInk는 팟캐스트 에피소드 제목 구문 일부로만 등장하며, 소프트웨어 제품으로는 미사용 확인.
- "설교필기"를 앱 이름으로 사용하는 제품 없음.
- 개발자 관점에서 실용적이고 앱 목적을 명확히 전달하는 이름으로 선택.

구현 계획:
- 상세 변경 대상 및 절차는 `docs/branding-rename-plan.md` 참조.
- 후속 세션에서 일괄 반영 예정.

## 2026-05-20: 태그 기반 GitHub Release 자동화

결정:
- Windows 배포는 `v*` 태그 push를 단일 릴리즈 트리거로 사용합니다.
- GitHub Actions가 Portable ZIP을 빌드하고 `SHA256SUMS.txt`를 생성한 뒤 GitHub Release에 첨부합니다.
- 태그 버전과 `pyproject.toml` 버전이 다르면 릴리즈를 중단합니다.

근거:
- 참고 프로젝트 `claude-usage-tray-windows`처럼 태그가 배포 의도를 명확히 표현합니다.
- GitHub Release 산출물과 checksum을 같은 자동화 경로에서 만들면 수동 업로드 누락을 줄일 수 있습니다.
- 버전 불일치 검증으로 앱 내부 버전, ZIP 파일명, 릴리즈 태그의 혼선을 방지합니다.

## 2026-05-20: 최근 작업 기록 비활성화(keep_history) 및 작업/캐시 연쇄 삭제 UX 도입

결정:
- 사용자 프라이버시 강화를 위해 최근 작업 기록 저장을 비활성화할 수 있는 `keep_history` 설정을 도입합니다.
- CLI 및 GUI 상에서 특정 작업을 삭제할 때, DB 레코드 삭제 뿐만 아니라 해당 작업의 오디오 전처리 파일 캐시 디렉터리(`cache/jobs/<job_id>`)도 연쇄적으로 안전하게 정리되도록 보강합니다.
- DB에 미등록된 채 방치된 잔여 작업 캐시 폴더들을 일괄 정리할 수 있도록 CLI `jobs clean-cache` 명령을 구현합니다.

근거:
- **개인정보 보호**: 로컬 데스크톱 프로그램이지만 설교자 및 강의 내용이 불필요하게 SQLite 데이터베이스나 최근 작업 목록에 영구적으로 쌓이는 것을 우려하는 사용자들을 위한 프라이버시 제어 기능이 필수적입니다.
- **안전한 데이터 관리**: 단순 DB 삭제 시 실제 오디오 전처리 캐시 파일이 지워지지 않아 디스크 낭비가 발생하므로, CASCADE 및 OS 디렉터리 연쇄 삭제 로직을 결합해 깔끔하게 정리합니다.
- **데이터 유실 방지**: 캐시 일괄 삭제 시 DB에 활성화된 ID와 철저히 비교하여, DB 상 실존하지 않는 '고스트 캐시'만 안전하게 필터링하여 일괄 삭제하는 안전 게이트를 적용했습니다.

## 2026-05-20: 오디오 싱크 플레이어 및 다중 배치 큐 연동

결정:
- 편집기(`TranscriptEditorWidget`) 내에 `PySide6.QtMultimedia` 기반의 오디오 싱크 플레이어 UI를 탑재하고 세그먼트 탐색(Seek) 및 하이라이팅 스크롤 연동.
- 메인 화면 대기 목록(`file_list`) 내 다중 파일 변환 시 순차 처리 기동 루프(`_start_next_queue_item`) 및 스레드 자원 경합 가드 탑재. 개별 실패 시에도 연속 변환을 처리하는 배치 큐 연쇄 기동 UX 구현.

근거:
- **오디오 싱크**: 편집 중 녹음 원본 또는 전처리 오디오를 즉시 탐색하고, 실시간 스크롤 하이라이팅을 통해 사용자가 편리하게 STT 수정 작업을 검수 및 편집하도록 유도.
- **안정적 다중 파일 처리**: 한 번에 한 작업씩 수행하는 워커 스레드가 확실히 종료된 후(`_on_thread_finished`)에 다음 대기열을 순차적으로 기동하게 함으로써 리소스 경합과 GUI 크래시를 원천 차단함.
