# Decision Log

## 2026-05-21: 프로 검수 단축키 및 Markdown 리치 편집 모드 도입 (v0.4.7)

결정:
- `TranscriptEditorWidget`에 마우스 제어가 필요 없는 `QShortcut` 기반 키보드 단축키 시스템(`Ctrl+Space`, `Ctrl+←/→`, `Ctrl+P`, `Ctrl+↑/↓`)을 도입하여 재생, 이동, 구간 재생, 배속 조절을 원스톱으로 제공합니다.
- 편집기 하단에 `QTextBrowser` 기반의 **Markdown 실시간 리치 텍스트 뷰어** 및 단축키 범례 안내 바를 탑재하고, 간이 Markdown 파싱 파이프라인을 내장해 실시간 스타일 서식을 피드백합니다.
- 사용자 제안 깃허브 이슈 #1("실시간 받아 적으면서 녹음이 되면 좋겠다")의 실시간 마이크 녹음 및 스트리밍 STT 아키텍처는 기술적 변경 범위와 의존성 추가 규모를 고려하여 다음 마일스톤(v0.5.0)의 최우선 단독 과제로 집중 이관하고 상세 인계 설계 가이드를 남깁니다.

근거:
- **검수 생산성의 비약적 향상**: 한영 타이핑 및 스페이스바 입력 등과 충돌을 막기 위해 `Ctrl+Space` 등의 보조 조합 핫키를 영리하게 활용하여 타이피스트들이 키보드에서 손을 떼지 않고 쾌속 검수하도록 지원합니다.
- **오프라인 렌더링 무결성**: 외부 렌더링 라이브러리 추가 없이 순수 정규식으로 안전하게 HTML 포맷으로 변환해 Markdown 실시간 뷰어를 제공하므로 경량화 및 스마트스크린 오탐 이슈를 원천 차단합니다.
- **안정적 마일스톤 격리**: 대규모 입출력 파이프라인 변화를 무리하게 진행하지 않고 v0.4.7 품질을 완벽히 매듭지어 배포한 뒤, 스트리밍 오디오 처리를 다음 차기 에이전트 세션의 핵심 설계로 온전히 격리 이관함으로써 소프트웨어의 릴리즈 무결성을 보장합니다.

## 2026-05-21: GitHub API 연동 실시간 업데이트 알리미 및 24시간 로컬 캐싱 체계 도입

결정:
- GitHub Releases API를 연동하여 최신 릴리즈 버전을 백그라운드 스레드(`UpdateCheckWorker`)에서 비동기 체크하는 알리미를 구현합니다.
- 익명 요청에 대한 API Rate Limit(시간당 60회) 초과 차단 및 부팅 딜레이를 방지하기 위해 `update_cache.json` 캐시 파일을 도입하여 24시간 캐싱(Bypass) 체계를 설계합니다.
- 수동 업데이트 확인(`force=True`)을 위한 도움말 메뉴 액션을 추가하고, 수동 시에는 캐시를 즉시 우회(Bypass)해 API 강제 조회를 수행한 뒤 QMessageBox 모달 대화창으로 명시적 피드백을 전달합니다.
- 딥 블루 그라데이션(`qlineargradient`)을 가미한 모던 프리미엄 `UpdateBannerWidget`을 GUI 상단에 유동적으로 표시하여 우수한 사용자 경험을 제공합니다.

근거:
- **안정적인 API 제어**: 사용자가 앱을 자주 재시작하더라도 로컬 캐시를 통하여 API 호출을 bypass하므로 Rate Limit 에러를 철저히 방지하며, 수동 확인 요구사항을 force 파라미터로 매끄럽게 흡수합니다.
- **보안 및 신뢰성**: SmartScreen 오탐 차단 및 보안 리스크 예방을 위해 자동 설치 프로그램 다운로드 방식 대신, 5초 timeout 및 User-Agent 헤더 필수 설정을 통해 안전하게 최신 공식 Release 브라우저 다운로드 페이지를 연동합니다.

## 2026-05-21: YouTube 입력 정책 갱신 — 고지 동의 기반 opt-in 유지

결정:
- v0.4.4 이후 구현된 YouTube URL 입력은 제거하지 않고, 저작권/이용 권한 고지 동의 기반 opt-in 기능으로 유지합니다.
- STT core는 계속 로컬 파일 파이프라인을 기준으로 두고, YouTube 처리는 `yt-dlp`를 통해 로컬 임시 오디오를 만드는 입력 소스 계층으로 취급합니다.
- 일반 온라인 다운로드, DRM/접근 제한 우회, 클라우드 업로드/동기화, 로그인·쿠키 기반 수집은 v1.0 범위에서 제외합니다.

근거:
- 현재 `main`에는 GUI 저작권 Disclaimer, 비동기 다운로드, yt-dlp 진단/설치 UI, 단위 테스트가 이미 포함되어 있고 v0.4.4/v0.4.5 릴리즈 이력에도 반영되어 있습니다.
- 문서가 과거 “v1.0 제외” 상태와 현재 구현 상태를 동시에 말하고 있어 사용자 안내와 릴리즈 검증이 충돌했습니다.
- 삭제보다는 권리 고지와 입력 계층 분리를 명문화해 기능의 위험 범위를 좁히는 것이 현재 저장소 상태에 더 적합합니다.

## 2026-05-20: 최근 작업 기록 비활성화(keep_history) 및 작업/캐시 연쇄 삭제 UX 도입

결정:
- 사용자 프라이버시 강화를 위해 최근 작업 기록 저장을 비활성화할 수 있는 `keep_history` 설정을 도입합니다.
- CLI 및 GUI 상에서 특정 작업을 삭제할 때, DB 레코드 삭제 뿐만 아니라 해당 작업의 오디오 전처리 파일 캐시 디렉터리(`cache/jobs/<job_id>`)도 연쇄적으로 안전하게 정리되도록 보강합니다.
- DB에 미등록된 채 방치된 잔여 작업 캐시 폴더들을 일괄 정리할 수 있도록 CLI `jobs clean-cache` 명령을 구현합니다.

근거:
- **개인정보 보호**: 로컬 데스크톱 프로그램이지만 설교자 및 강의 내용이 불필요하게 SQLite 데이터베이스나 최근 작업 목록에 영구적으로 쌓이는 것을 우려하는 사용자들을 위한 프라이버시 제어 기능이 필수적입니다.
- **안전한 데이터 관리**: 단순 DB 삭제 시 실제 오디오 전처리 캐시 파일이 지워지지 않아 디스크 낭비가 발생하므로, CASCADE 및 OS 디렉터리 연쇄 삭제 로직을 결합해 깔끔하게 정리합니다.
- **데이터 유실 방지**: 캐시 일괄 삭제 시 DB에 활성화된 ID와 철저히 비교하여, DB 상 실존하지 않는 '고스트 캐시'만 안전하게 필터링하여 일괄 삭제하는 안전 게이트를 적용했습니다.
